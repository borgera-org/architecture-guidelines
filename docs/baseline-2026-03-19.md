# Baseline 2026-03-19

## Proposito

Este documento registra o baseline atual da plataforma de arquitetura no estado
de 2026-03-19.

Ele existe para consolidar as decisoes estruturais ja tomadas, reduzir
dependencia do historico da conversa e servir como checkpoint operacional para a
proxima etapa de evolucao do repositorio.

## Resumo Executivo

No estado atual, este repositorio ja deixou de ser apenas uma base de
documentacao.

Ele ja funciona como:

- fonte de verdade para humanos
- fonte de contexto estavel para IA
- backend contratual para um CLI interno de scaffolding

Ja existe um primeiro archetype oficial em estado `active`, um conjunto inicial
de templates, schemas para contratos principais, tres examples materializados,
validadores automatizados e uma primeira implementacao de referencia do CLI.

## Papel do Repositorio

Este repositorio foi definido como a fonte oficial para:

- diretrizes de arquitetura reutilizaveis
- governanca de evolucao dos padroes
- archetypes de repositorio

- examples de referencia

quando virar padrao transversal reutilizavel.

## Decisoes Estruturais Ja Tomadas

As decisoes abaixo ja devem ser tratadas como baseline da plataforma.

### 1. Este repositorio e source of truth

O repositorio foi posicionado explicitamente como fonte oficial para
desenvolvedores, IA e CLI.

### 2. A plataforma foi separada em camadas

As camadas adotadas foram:

- `docs/` para explicacao e governanca
- `archetypes/` para definicao do tipo de repositorio
- `templates/` para materializacao
- `schemas/` para validacao
- `examples/` para saida esperada
- `scripts/` para verificacoes automatizadas

### 3. O primeiro archetype oficial e `api-dotnet`

Esse archetype ja existe, foi promovido para `active` e representa o baseline
oficial atual para APIs HTTP em `.NET`.

### 4. Ja existe uma implementacao inicial do CLI

O repositorio agora ja contem uma implementacao de referencia em `cli/` que
consome o contrato atual do archetype `api-dotnet`.

### 5. `templateSet` suporta selecao condicional com subset formal de `when`

O contrato atual ja suporta templates condicionais com `when`, por exemplo:

- `test-project` quando `includeTests == true`
- `docker-files` quando `includeDocker == true`

No baseline atual, `when` foi formalizado como comparacao booleana simples no
formato `<booleanInput> == true|false`.

### 6. Templates renderizaveis usam `.tpl`

Arquivos com `.tpl` devem ser renderizados pelo CLI e materializados sem o
sufixo no destino.

Arquivos sem `.tpl` devem ser copiados literalmente.

### 7. O CLI precisa de pos-processamento

O CLI nao pode funcionar apenas como mecanismo de copia de arquivos.

No baseline atual, ele ja precisa:

- adicionar o projeto principal a solution
- adicionar o projeto de testes a solution quando aplicavel

Esse comportamento agora ja esta publicado de forma declarativa no bloco
`postProcessing` do archetype `api-dotnet`.

### 8. Os examples oficiais ja cobrem o caminho padrao e as variacoes condicionais principais

Os snapshots materializados de `billing-api` hoje cobrem:

- o caminho padrao em `minimal-default`
- a variacao sem projeto inicial de testes em `without-tests`
- a variacao sem artefatos de Docker em `without-docker`

## Convencoes Ja Adotadas

### Identidade tecnica

- `rootNamespace` representa a identidade tecnica do projeto `.NET`
- `serviceName` representa melhor a identidade logica do servico

### Projeto principal

- o caminho inicial do projeto principal usa `src/{{ rootNamespace }}`
- o nome do `.csproj` principal usa `{{ rootNamespace }}`

### Projeto de testes

- o caminho inicial do projeto de testes usa `tests/{{ rootNamespace }}.Tests`
- o framework inicial adotado e `xUnit`
- o baseline de teste inicial valida `/health` com `WebApplicationFactory`

### Solution

- a solution nasce vazia no template
- o CLI deve conectar projetos a ela via pos-processamento

### Docker

- o `Dockerfile` usa build multi-stage
- a tag base das imagens `.NET` e controlada por `dotnetImageTag`

1. iniciar o proximo archetype oficial

## Regra Pratica

Se uma decisao importante da plataforma ja tiver sido tomada, ela deve ser
refletida em contrato, schema, example ou baseline.

Se ela existir apenas no historico da conversa, ainda nao esta suficientemente
consolidada.
